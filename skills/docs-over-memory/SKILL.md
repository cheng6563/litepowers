# 先查文档，别信记忆

训练记忆对**版本相关的事实**天然过时：API 改签名、CLI 换 flag、配置键改名、默认值变更、依赖行为随大版本漂移。凭记忆答，看着自信，其实是猜。

**核心：给出版本敏感的断言前，先用官方源核对当前版本。**

## 触发（先核对再答）

- 工具 / 库 / 框架的**版本特定行为**、默认值、弃用；
- **API / CLI 用法**：签名、参数、flag、返回结构；
- **配置**：键名、格式、生效范围；
- 依赖升级后的行为变化；
- 用户点明「最新」「当前」「这个版本」或时效性。

## 跳过（记忆足够）

- 稳定的语言基础语法、通用算法、设计原则；
- 用户已给定版本和确切行为；
- 纯本仓事实（读代码 / 配置 / lockfile 更准，不必外搜）。

先分清「这是稳定常识还是版本敏感事实」——拿不准就当版本敏感,去核对。

## 怎么核对

1. **优先官方源**：官方文档、正式 changelog、release notes、仓库源码 / 类型定义；本仓行为以代码和 lockfile 为准。
2. **对齐版本**：先确认项目实际用的版本（lockfile / 已装版本），再查那个版本的文档，不是查「最新」就完事。
3. **评估来源**：警惕为 AI 引用优化的低质 GEO 内容、内容农场、过时博客和版本不明的答案；多个权威源一致才当强证据。
4. **区分证据强度**：官方文档 / 源码为强，多处一致的社区用法为中，单篇博客或直觉为弱；弱证据要标注不确定，不写成事实。

## 与相邻 skill 边界

- 本 skill 管**下结论前的外部事实核对**（文档/源码怎么说）；`skill:verification` 管**跑命令拿新鲜执行证据**（实际跑出来怎样）。前者读文档，后者看输出，互不替代。
- 能直接运行验证的，跑一遍比读文档更硬——但运行不便或属通用知识断言时，官方文档核对是最低标准。

## 雷区

- 别把过时记忆包装成确定语气；不确定就说不确定，去查。
- 别查了「最新版」就套到用户的**旧版本**上。
- 别拿单篇搜索结果当权威；先掂来源可靠性。
