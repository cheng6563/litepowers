# Align：高风险动手前对齐

复杂任务先降低错误方向的风险；明确、低风险的小改动不要被仪式阻塞。

## 先分级

进入完整对齐，只要存在一项：

- 需求或验收含糊；
- 跨模块或影响外部接口；
- 多个方案有实质取舍；
- 架构显著、难回滚或迁移成本高；
- 涉及安全、权限、数据归属或兼容性；
- 无法用一句话描述预期 diff 和 done-when。

若任务低风险、方案已由用户明确、预期 diff 与完成判据清楚：简短复述 `done-when` 后直接执行，不强制再给 2–3 个方案或重复索取批准。

## 完整对齐的最小产物

```text
Goal
Non-goals
Constraints
Done-when
Delivery target（本次做到哪一步）
Consumes（复杂任务可选）
Produces（复杂任务可选）
```

## 流程

1. **探索事实**：读相关代码、文档、已有决策、真实调用方和最近改动，不在真空里提问。
2. **逐个消除关键未知**：优先问会改变实现路线的问题；一轮可包含少量互相独立的问题，避免倾倒。
3. **给候选方案**：存在真实取舍时给 2–3 个方案，推荐项置顶，说明成本、风险和约束；没有真实替代方案就不要凑数。
4. **确认完成标准**：把结果写成可观察、可验证的 `done-when`。
5. **确认交付终点**：询问“本次工作做到哪一步”，并记入 `Delivery target`；用户已明确时只复述。只提供由项目事实证明可达的层级：实现并验证、提交、推送、合并或测试/生产部署；不猜远端、目标分支或部署能力，并说明所选终点的必要门禁。
6. **获得关键决策后执行**：不为低价值细节反复索取批准。

范围包含多个独立子系统时，先拆成可独立验收的子任务；不要一次设计整个平台。

## 平台增强

对有限且会改变路线的决策使用结构化单选，推荐项置顶并说明理由；代码、ASCII 布局或结构差异可提供预览。开放问题或工具不可用时使用简洁文字问答。适用选项超过工具上限时改用编号列表，不隐藏选项。

## 原则

- 风险越高，对齐越充分；风险越低，越应直接行动。
- 先查既有约束，再提出方案。
- YAGNI：砍掉成功标准未要求的能力。
- 方案要反映真实取舍，不制造虚假选择。
- 对外行为和跨模块接口要检查权限、状态、租户、数据归属、幂等和生命周期语义。
- 对齐结论落入代码、测试或项目已有文档体系；不自动创建独立重型计划或 ADR。
