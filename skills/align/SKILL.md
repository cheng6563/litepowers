# Align：高风险动手前对齐

复杂任务先降低错误方向的风险；明确、低风险的小改动不要被仪式阻塞。

## 先分级

进入完整对齐，只要存在一项：

- 需求或验收含糊；
- 跨模块或影响外部接口；
- 多个方案有实质取舍；
- 架构显著、难回滚或迁移成本高；
- 涉及安全、权限、数据归属或兼容性；
- 无法用一句话描述预期 diff 和 done-when。

若任务低风险、方案已由用户明确、预期 diff 与完成判据清楚：简短复述 `done-when` 后直接执行，不强制再给 2–3 个方案或重复索取批准。

## 完整对齐的最小产物

```text
Goal
Non-goals
Constraints
Done-when
Delivery target（本次做到哪一步）
Consumes（复杂任务可选）
Produces（复杂任务可选）
```

## 流程

1. **探索事实**：读相关代码、文档、已有决策、真实调用方和最近改动，不在真空里提问。
2. **逐个消除关键未知**：优先问会改变实现路线的问题；一轮可包含少量互相独立的问题，避免倾倒。
3. **给候选方案**：存在真实取舍时给 2–3 个方案，推荐项置顶，说明成本、风险和约束；没有真实替代方案就不要凑数。
4. **确认完成标准**：把结果写成可观察、可验证的 `done-when`。
5. **确认交付终点**：询问“本次工作做到哪一步”，不要把“代码完成”和“交付完成”混为一谈。先探测当前项目真实具备的能力，再按递进顺序给出适用选项：
   - **实现并验证成功**：始终提供；验证方式按改动类型选择，不强行等同于单元测试。
   - **提交**、**推送当前分支**、**合并项目默认分支并推送**：仅在 Git 仓库中提供；没有远端时不提供推送，没有可确定的目标分支时不猜 `main` / `master`。
   - **部署到测试环境**、**部署到生产环境**：仅当项目已有可执行且可验证的对应部署方案时提供；测试与生产能力分别判断，不因存在 Dockerfile、CI 文件或模糊描述就推定可部署。
   - 只展示实际可执行的层级；若高层选项成立，默认包含此前必要步骤，并在选项描述中说清楚。例如“推送当前分支”包含实现、验证和提交。
   - 把用户选择记入 `Delivery target`。用户原请求已经明确终点时，复述确认即可，不重复询问。
6. **获得关键决策后执行**：不为低价值细节反复索取批准。

范围包含多个独立子系统时，先拆成可独立验收的子任务；不要一次设计整个平台。

## Claude Code 平台增强

有 `AskUserQuestion` 时，可用它呈现真正需要用户决定的 2–4 个选项：

- 推荐项放第一并标注 `(推荐)`；
- 代码、ASCII 布局或结构差异可放 `preview`；
- 开放探索或内容过长时改用普通文字；
- 只在决定会改变后续工作时询问，不用选择器确认常规默认值。
- 交付终点通常用单选；适用终点不超过 4 个时用选择器，超过 4 个时改用简洁的普通编号列表，不能为迁就工具上限而隐藏可执行层级。

其他 Agent 没有该工具时，直接用简洁问答实现相同行为，不提平台专有工具名。

## 原则

- 风险越高，对齐越充分；风险越低，越应直接行动。
- 先查既有约束，再提出方案。
- YAGNI：砍掉成功标准未要求的能力。
- 方案要反映真实取舍，不制造虚假选择。
- 对外行为和跨模块接口要检查权限、状态、租户、数据归属、幂等和生命周期语义。
- 对齐结论落入代码、测试或项目已有文档体系；不自动创建独立重型计划或 ADR。
- 对齐时选择交付终点，是对本次后续动作的范围授权；执行到推送、合并、部署等外部动作前仍要核对前置验证、目标分支/环境和项目既有安全门。用户选择的终点不授权绕过保护规则，也不授权未列明的发布、删改或迁移动作。

## 不做什么

- 不对所有创意工作设置统一硬门。
- 不在用户已明确方案时重新让用户选择同一件事。
- 不用一大段文字同时倾倒问题、方案和实现细节。
- 不在关键需求仍未知时开始不可逆实现。
- 不把本地存在的容器、工作流或脚本名称当作部署能力的充分证据。
- 不在非 Git 项目里询问提交/推送，也不在缺少对应发布方案时诱导用户选择测试或生产部署。
- 不把“合并默认分支”硬编码成合并 `main` 或 `master`。
