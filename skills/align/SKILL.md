# Align：高风险动手前对齐

复杂任务先降低错误方向的风险；明确、低风险的小改动不要被仪式阻塞。

## 先分级

进入完整对齐，只要存在一项：

- 需求或验收含糊；
- 跨模块或影响外部接口；
- 多个方案有实质取舍；
- 架构显著、难回滚或迁移成本高；
- 涉及安全、权限、数据归属或兼容性；
- 无法用一句话描述预期 diff 和 done-when。

若任务低风险、方案已由用户明确、预期 diff 与完成判据清楚：简短复述 `done-when` 后直接执行，不强制再给 2–3 个方案或重复索取批准。

## 完整对齐的最小产物

```text
Goal
Non-goals
Constraints
Done-when
Delivery target（本次做到哪一步）
Consumes（复杂任务可选）
Produces（复杂任务可选）
```

## 流程

1. **探索事实**：读相关代码、文档、已有决策、真实调用方和最近改动，不在真空里提问。
2. **逐个消除关键未知**：优先问会改变实现路线的问题；一轮可包含少量互相独立的问题，避免倾倒。
3. **给候选方案**：存在真实取舍时给 2–3 个方案，推荐项置顶，说明成本、风险和约束；没有真实替代方案就不要凑数。
4. **确认完成标准**：把结果写成可观察、可验证的 `done-when`。
5. **确认交付终点**：询问“本次工作做到哪一步”，并记入 `Delivery target`；用户已明确时只复述，不重复询问。按项目已确认的能力递进提供：
   - **实现并验证成功**：始终提供；
   - **提交 / 推送当前分支 / 合并默认分支并推送**：依次要求 Git 仓库、可用远端、可确定的目标分支；不猜 `main` / `master`；
   - **部署测试环境 / 生产环境**：分别要求项目已有可执行、可验证的对应方案；Dockerfile 或含糊 CI 不算充分证据。
   高层选项应说明必要前置步骤；选择终点不代表可以绕过验证、分支保护或部署安全门。
6. **获得关键决策后执行**：不为低价值细节反复索取批准。

范围包含多个独立子系统时，先拆成可独立验收的子任务；不要一次设计整个平台。

## Claude Code 平台增强

有 `AskUserQuestion` 时，可用它呈现真正需要用户决定的 2–4 个选项：

- 推荐项放第一并标注 `(推荐)`；
- 代码、ASCII 布局或结构差异可放 `preview`；
- 开放探索或内容过长时改用普通文字；
- 只在决定会改变后续工作时询问，不用选择器确认常规默认值。
- 交付终点用单选；超过 4 个适用选项时改用普通编号列表，不隐藏选项。

其他 Agent 没有该工具时，直接用简洁问答实现相同行为，不提平台专有工具名。

## 原则

- 风险越高，对齐越充分；风险越低，越应直接行动。
- 先查既有约束，再提出方案。
- YAGNI：砍掉成功标准未要求的能力。
- 方案要反映真实取舍，不制造虚假选择。
- 对外行为和跨模块接口要检查权限、状态、租户、数据归属、幂等和生命周期语义。
- 对齐结论落入代码、测试或项目已有文档体系；不自动创建独立重型计划或 ADR。

## 不做什么

- 不对所有创意工作设置统一硬门。
- 不在用户已明确方案时重新让用户选择同一件事。
- 不用一大段文字同时倾倒问题、方案和实现细节。
- 不在关键需求仍未知时开始不可逆实现。
