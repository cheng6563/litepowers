# 代码即业务

业务的唯一真相在**代码**里。代码流程就是业务流程；文档只做**业务→代码的索引**，不复述逻辑。

## 三层载体

| 载体 | 放什么 | 为什么 |
|---|---|---|
| 代码 / SQL / 配置 | 业务流程、规则、状态流转、约束本身 | 会被执行，不会骗人也不会漂 |
| 就近注释 | 局部的 why、决策点、非显然取舍、待确认项 | 跟代码同处一地，一起改，天然不漂 |
| 文档 | 业务名/流程/边界 → 代码位置的索引，跨模块全景，ADR 级决定 | 代码看不到的全局视角 |

独立的需求文档、业务说明文档是**反模式**：它离代码越远越会漂，且 AI 读代码时根本不会去翻。要写的业务，先想能不能变成更清楚的代码 + 就近注释。

## 黄金判据

> 删掉这段文字，AI 读**代码 + 注释**能不能还原出来？
> **能 → 不写；不能 → 先塞进就近注释；注释放不下的跨模块/长期视角 → 才写进文档，且只写成索引。**

“为什么这么写”的局部 why 属于注释；长期架构决定是否需要 ADR，按 `skill:decision-layering` 的门槛判断。

## 不归它管：风格与约定

本 skill 只管**业务**该放哪，不反对写团队约定。代码风格、命名规范、分层/错误处理约定不是业务，别塞进业务索引文档，也别塞注释：

- **能机械判定**（格式、缩进、引号、import 顺序、命名模式）→ formatter / lint / editorconfig / CI。写成文档没人执行；只有门禁能真正拦住违反。
- **要 AI 遵守、lint 判不了**（错误处理策略、分层规则、事务/并发约定）→ 放 AI 每次自动加载、优先级最高的入口规则（CLAUDE.md / AGENTS.md），不要放独立文档。理由和业务一样：独立文档 AI 不主动翻，入口规则才会被强制加载、每次遵守。

黄金判据对它们自洽：「删掉后 AI 读代码还原不出来」→ 判定保留；反对的只是复述代码能推断的东西，和把业务写进远离代码的文档。承载层的通用选择见 `skill:decision-layering`。

## 让代码配得上当业务读

代码乱 → 业务读不出 → 被迫写文档补救，是恶性循环。改动代码时顺手清理这四维（**只看整洁，不查 correctness bug——那是 `skill:code-review` 的事**）：

- **复用**：新代码是否重造了已有 helper？先 grep 共享模块和相邻文件，改成调用已有的。
- **简化**：删冗余或可推导的状态、复制粘贴的变体、过深嵌套、遗留死代码；命名要能读出业务意图。
- **效率**：去掉重复计算和重复 I/O；独立操作别串行；别往启动或热路径塞阻塞逻辑。长生命周期对象别从闭包捕获整个环境，只拷需要的字段。
- **高度**：改动打在正确的深度，别在共享设施上层层叠特例；宁可把底层机制泛化，也不加一次性补丁。

改法不改变原意、不外溢到本次改动之外时直接改；会动到意图或范围就先说明。

## 清理 vs 保留

| 该清理（代码/注释可还原） | 该保留（放注释或文档索引） |
|---|---|
| 字段类型、约束、默认值、索引 | 业务规则、状态机决策点 → 就近注释 |
| 接口参数和响应字段清单 | 枚举的业务语义、审批链 → 就近注释 |
| 纯实现步骤、调用顺序 | 跨模块联动、设计取舍及原因 → 注释 + 文档索引 |
| 与 schema、DTO、接口定义重复的内容 | 待确认项、证据强度、为什么这么定 |

## 写/改文档的边界

1. **只写索引，不复述逻辑**：文档指向代码位置和业务名，不抄字段类型、参数清单、调用顺序（以代码为准）。
2. **业务优先进注释**：能写进就近注释的业务规则/不变量，不要单独起一篇文档；文档只在需要跨模块全景时补一层索引。
3. **证据强度不升级**：用户口述或未实操确认的写操作要明确标注，不擅自写成已验证事实。
4. **不复制政策**：ADR 生命周期等通用政策引用对应规则，不在多篇重复定义。

## 检查漂移

索引和注释仍会漂。逐处核对实现：

| 类型 | 例 |
|---|---|
| 旧概念残留 | 已迁移配置键、已删字段、废弃写法 |
| 可推断重复 | 文档抄了字段类型、参数清单、调用顺序 |
| 过时引用 | 旧包路径、删除类名、改名方法、失效链接 |
| 遗漏规则 | 代码已有但注释/文档没说明的业务不变量 |
| 证据漂移 | “未确认”被改成“已确认” |
| 注释脱节 | 注释描述的行为和相邻代码已经不一致 |

## 工作流

**增量编辑**：先搜重复，读目标文档/注释和相关实现；范围不明确或删除可能损失业务知识时，先报告疑点再改，拿不准标 `[需确认]`。

**批量审计**：先列清单和修正方向；逐处核对实现，只修漂移和可清理的重复，不借机扩写或重构无关内容。是否分提交服从用户和项目约定，不自行 commit。

**存量文档迁移**（老项目一堆 md）：不推倒重写，先分清读者——「AI 不看」只对**给 AI 当上下文**的那部分成立；onboarding、外部集成、合规留痕仍有人类读者，不能以此为由删。分类分级、增量迁移：

| 现状 | 处置 |
|---|---|
| 纯可推断（字段表、API 清单、抄 schema） | 瘦成指向代码的索引；能自动生成的交给生成器 |
| 业务 why 散落文档 | 逐块下沉到就近注释——大改，一块一块确认，别一次性搬 |
| 风格 / 约定 | 迁 lint / CLAUDE.md，见上文 |
| ADR / 跨模块全景 | 留着，精简成索引 |
| 过时 / 死文档 | 标记待删，用户拍板 |

先出清单和方向，小步走，不自动搭迁移工程，不自行 commit。

## 雷区

- 修漂移不等于重写文档；清理代码不等于顺手改行为。
- 别把该进代码/注释的业务塞进独立文档，也别把通用常识塞进项目文档。
- 不查 correctness bug，那是 `skill:code-review`。
- 不静默重写 ADR 的实质决策；维护性修正和 supersede 规则见 `skill:decision-layering`。
