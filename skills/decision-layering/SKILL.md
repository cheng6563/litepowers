# 决策分层：新规则该落哪一层

一条新约束先选择**最小必要承载层**。解释、发现和执行可以分层，但不要默认每层都写一遍。

## 先查已有规则

先检查入口规则、ADR、模块文档、测试、lint、hook 和 CI：

- 已有规则没被看见 → 改善可发现性或引用路径；
- 已有机械规则没执行 → 补确定性的检查；
- 规则不存在 → 再选择承载层；
- 老项目没有 ADR/spec → 先从调用方、数据结构、兼容行为和线上约束还原事实，别把猜测固化。

## 各层职责

| 层 | 适合保存 |
|---|---|
| ADR | 架构显著、长期、难逆转且理由无法从代码恢复的决定与取舍 |
| Spec / 模块文档 | 业务名/流程/边界 → 代码位置的索引，跨模块全景；业务规则和状态机本身优先落代码+就近注释，文档不复述 |
| CLAUDE.md / AGENTS.md | 稳定、高频、普遍适用的项目规则 |
| Skill | 偶发、专项、跨项目复用的工作方法 |
| Test / lint / type / CI | 可机器判断的约束与结果 |
| Hook | 必须在特定生命周期发生的动作或快速反馈 |
| 代码注释 | 局部且无法从代码本身恢复的 why |
| Issue / commit | 一次性变更的上下文和短期理由 |

## 决策顺序

1. **它是当前实现事实吗？** 放代码、schema 或配置，不重复写文档。
2. **它是产品行为或业务不变量吗？** 规则本身落代码 + 就近注释，可判定部分用测试固化；文档只在需要跨模块全景时补一层索引。载体细节见 `skill:code-as-spec`。
3. **它是可机械判断的规则吗？** 优先 lint/type/test/CI；需要编辑期反馈再考虑 hook。
4. **它是工作方法吗？** 跨项目复用才写 Skill；项目特有步骤放项目文档。先查现有 skill：已有 skill 覆盖同职责就增强或合并，别新增重叠的（重叠会让触发互抢）。确定写 skill 后，怎么写好见 `skill:writing-skills`。
5. **它是架构决定吗？** 仅同时具有长期或高影响、存在真实替代方案、迁移代价较高、理由难从代码恢复等特征时考虑 ADR。
6. **一次性或局部取舍是否已能由注释、issue 或 commit 充分保存？** 能就到此为止。

一个规则可以有“解释层 + 执行层”，例如模块文档解释权限语义、测试强制权限结果；但每增加一层都要回答它提供了什么独立价值。

## ADR 门槛

适合 ADR：

- 模块、服务或数据所有权边界；
- 持久化、协议、消息机制、关键平台或长期依赖选择；
- 安全、合规、可靠性策略；
- 多个合理方案间有重要权衡；
- 难逆转、迁移成本高，且 why 仅看代码难恢复。

不适合 ADR：普通 bugfix、局部实现、命名格式、机械规则、无真实替代方案的例行变更，以及 commit/issue/注释已足够的内容。

项目没有 ADR 体系时，不要为一条边缘决定自动搭目录、编号和模板；先说明价值与维护成本，由用户决定是否引入。已有体系则遵循现有格式。

### ADR 生命周期

不要静默重写已经生效的**实质决策**：决策、适用范围、关键理由或权衡实质改变时，新建 superseding ADR，并按项目约定关联旧记录。

以下维护性修改可原地进行：typo、坏链接、格式/元数据、明显事实错误、不改变含义的澄清，以及敏感信息清除；需要审计时保留变更记录。

## Enforcement 选择

- 高代价且可稳定判定 → CI/test/type/lint 应成为最终门禁；hook 可提供更快反馈。
- 语义规则无法机械完整判断 → 文档解释 + 针对性测试 + review，不要伪造“万能正则”。
- 阻塞式 hook 只用于判定足够稳定、误报可接受且违反代价高的规则；否则使用非阻塞反馈或只在 CI 拦截。
- prompt 式 AI hook 不是默认方案；优先确定性机制。

## 自检

1. 最小承载层是哪一个？
2. 是否重复了代码或另一份文档？
3. 哪部分可机械判断，放在哪里执行？
4. AI 在需要时能否发现它？
5. ADR 门槛是否真的满足？
6. 新增层的独立价值是否大于漂移成本？

## 跳过

普通 bugfix、既定约定内实现、原型、临时脚本、只影响单模块且模块文档足够的内部规则，不需要启动完整分层分析。
