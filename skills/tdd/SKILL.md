# TDD：先写测试

先写测试，看它失败，再写最小实现让它通过。

**核心：没看着测试失败过，就不知道它测的是不是对的东西。**

## 铁律

```
没有先失败的测试，不写生产代码。
```

先写了实现才补测试？删掉重来。不留作"参考"、不"改改用"、不看它——删就是删，从测试重新实现。

## 老项目没有单测框架时

TDD 的核心是“先有可失败证据”，不是“必须单元测试”：

- 优先用项目已有验证入口：集成测试、CLI、脚本、页面冒烟、录制回放、最小复现命令。
- 没有测试框架 → 写最小 characterization / regression 脚本，或用现有运行方式构造可重复失败步骤。
- 框架太旧/太重跑不了 → 先保留手动复现步骤 + 期望输出 + 实际失败证据，再做最小实现；同时标注“无自动覆盖”的原因。
- 别为了 TDD 大改测试基建；测试基建本身是独立工作，需要用户同意。

## 红-绿-重构

```dot
digraph tdd {
    rankdir=LR;
    red [label="RED\n写失败测试", shape=box];
    vr [label="看它正确地失败", shape=diamond];
    green [label="GREEN\n最小实现", shape=box];
    vg [label="看它通过\n全绿", shape=diamond];
    refactor [label="REFACTOR\n清理", shape=box];

    red -> vr;
    vr -> green [label="对"];
    vr -> red [label="失败原因不对"];
    green -> vg;
    vg -> refactor [label="对"];
    vg -> green [label="没过"];
    refactor -> vg [label="保持绿"];
}
```

**RED** — 写一个最小测试，描述应该发生什么。一个行为、清晰命名、用真实代码（非必要不 mock）。

**验证 RED（必做，别跳）** — 跑测试，确认它**失败**（不是报错），且失败原因是功能缺失（不是拼写错）。测试直接过了？说明你在测已有行为，改测试。

**GREEN** — 写**最简**代码让它过。别加测试没要求的功能、别顺手重构、别过度设计（YAGNI）。

**验证 GREEN（必做）** — 跑测试，确认通过、其他测试仍绿、输出干净（无报错警告）。没过 → 改代码不改测试。

**REFACTOR** — 只在绿之后：去重、改名、抽 helper。保持绿，不加新行为。

## 例外（需用户同意）

一次性原型 / 生成代码 / 配置文件。想"就这次跳过 TDD"？停，那是合理化借口。

## 红旗——出现就删掉重来

- 先写实现后补测试
- 测试一写就过
- 说不清测试为什么失败
- 为了一个小改动先大改测试基建
- "已经手动测过了"
- "测后面再补"
- "这次不一样，因为……"

## 卡住时

| 问题 | 解法 |
|------|------|
| 不知道怎么测 | 先写理想中的 API 调用 / 先写断言 |
| 测试太复杂 | 是设计太复杂，简化接口 |
| 什么都得 mock | 代码耦合太重——依赖写死在内部，改成从构造器注入 |
| 全是 mock、只在验证调用顺序 | 别硬单测。外部副作用收进薄 adapter：以内单测；adapter 本身按条件选最轻验证（内存替身/录制回放/手动冒烟），没条件就显式标注"无自动覆盖"，别堆 mock 假绿 |

bug 修复同理：先写复现该 bug 的失败测试，再按红-绿-重构修——测试既证明修好了又防回归。
