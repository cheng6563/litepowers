# 完成前验证

没验证就说完成，是不诚实，不是高效。

**核心：先证据，后结论，永远如此。**

## 铁律

```
没有刚跑出来的验证证据，不下"完成 / 通过"的结论。
```

不是在这条消息里跑过验证命令，就不能说它通过。

## 闸门

下任何"完成 / 通过 / 修好"结论、表达任何满意（"搞定！""完美！"）之前：

1. **想清楚**：什么命令能证明这个结论？
2. **跑**：完整跑一遍（新鲜的、完整的，不是上次的）。
3. **读**：看完整输出、查退出码、数失败数。
4. **核对**：输出真的支持结论吗？不支持就如实报实际状态。
5. **才**下结论，且**带上证据**。

跳过任一步 = 撒谎，不是验证。

## 常见错配

| 结论 | 需要的证据 | 不够格的 |
|------|-----------|----------|
| 测试通过 | 测试命令输出：0 失败 | 上次跑的 / "应该过" |
| Lint 干净 | linter 输出：0 错误 | 抽查 / 推断 |
| 构建成功 | 构建命令：退出 0 | "lint 过了" |
| Bug 修好 | 测原始症状：通过 | 改了代码就假设好了 |
| 防回归测试有效 | 红-绿循环验证过 | 测试过了一次 |
| 子 agent 完成了 | 看 VCS diff 确认改动 | agent 自报"成功" |
| 需求达成 | 逐条对照清单 | "测试过了" |

## 红旗——停

- 用"应该""大概""看起来"
- 验证前就表达满意
- 没验证就准备 commit / push / PR
- 信子 agent 的"成功"自报
- 靠抽查 / 局部验证
- "就这一次""我很有信心"——信心不是证据

## 底线

跑命令。读输出。**然后**才下结论。不容讨价还价。
