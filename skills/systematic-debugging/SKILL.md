# 系统化调试

乱试浪费时间还引入新 bug，快速打补丁掩盖真问题。

**铁律：动手修之前先定位根因。** 修症状 = 失败。

## 四阶段

逐阶段推进，前一阶段没完成不进下一阶段。

### 1. 定位根因（动手前必做）

- **认真读报错** — 别跳过，错误信息常常就含答案。读完整栈、行号、文件、错误码。
- **稳定复现** — 能可靠触发吗？步骤是什么？不能复现就再收集数据，别猜。
- **看最近改动** — git diff / 最近 commit / 新依赖 / 配置变更。
- **多组件系统先加埋点** — 系统有多层（请求→服务→数据库 等）时，在每个边界打日志（进什么、出什么、配置有没有透传），**跑一次拿证据看在哪一层断**，再去查那一层。
- **顺数据流回溯** — 坏值从哪来？谁传进来的？一路往上追到源头，在源头修，不在症状处修。

### 2. 找模式

- 找同代码库里类似的**能跑**的例子。
- 要照搬某个范式，就**完整读**参考实现，别只扫一眼。
- 列出能跑的和坏的之间**每一处差异**，别假设"这个不可能有影响"。

### 3. 假设与验证

- **单一假设**：明确写下"我认为 X 是根因，因为 Y"。
- **最小测试**：做能验证假设的最小改动，一次只动一个变量。
- **先验证再继续**：成了 → 阶段 4；没成 → 提新假设，**别在旧改动上叠新补丁**。
- **不懂就说不懂**，别装懂，去查或问。

### 4. 实现

- **先写复现测试**（可借 `tdd` skill），有了再修。
- **单一修法**，一次一个改动，不夹带"顺手优化"。
- **验证**：测试过了？没弄坏别的？问题真解决了？
- **修不好就停**：数一下试了几次。≥3 次失败别再试第 4 个补丁——**这通常是架构问题**，停下来和用户讨论是不是范式本身错了（每个补丁都在新地方冒出新症状 = 架构信号，不是假设失败）。

## 红旗——出现就停，回阶段 1

- "先快速修一下，回头再查"
- "改改 X 看看行不行"
- "一次多改几处再跑测试"
- "跳过测试，我手动验"
- "大概是 X，修了再说"
- "不太懂但这样可能行"
- 还没追数据流就开始提解法
- 已经试了 2+ 次还要"再试一个"

## 真没有根因时

系统化排查后确认确实是环境 / 时序 / 外部因素导致：记录排查过程，加合适的处理（重试 / 超时 / 明确报错）+ 监控日志。但 95% 的"没有根因"是排查不彻底。
