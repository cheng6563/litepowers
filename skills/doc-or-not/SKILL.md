# 文档该不该写

文档只保留**代码 / SQL / 配置无法充分表达**的知识。

## 黄金判据

> 删掉这段，AI 读代码能不能推断出来？**能 → 通常不写；不能 → 选择最贴近读者的载体。**

“为什么这么写”的局部细节可放代码注释；长期架构决定是否需要 ADR，按 `skill:decision-layering` 的门槛判断。

## 清理 vs 保留

| 该清理（代码可推断） | 该保留（代码读不出） |
|---|---|
| 字段类型、约束、默认值、索引 | 业务规则、状态机决策点 |
| 接口参数和响应字段清单 | 枚举的业务语义、审批链 |
| 纯实现步骤、调用顺序 | 跨模块联动、设计取舍及原因 |
| 与 schema、接口定义、DTO 重复的内容 | 待确认项、证据强度、为什么这么定 |

## 写新文档 / 改文档的边界

1. **字段表**：只补代码难表达的业务语义和分组；类型、约束、默认值以 schema 为准。状态/枚举值散落且语义不明显时可集中说明。
2. **证据强度不升级**：用户口述或未实操确认的写操作应明确标注，不能擅自写成已验证事实。
3. **不复制政策**：ADR 生命周期等通用政策引用对应规则，不在多篇文档重复定义。

## 检查漂移

逐篇检查：

| 类型 | 例 |
|---|---|
| 旧概念残留 | 已迁移配置键、已删字段、废弃写法 |
| 可推断重复 | 字段类型、参数清单、调用顺序 |
| 过时引用 | 旧包路径、删除类名、改名方法 |
| 遗漏规则 | 代码已有但文档没说明的业务不变量 |
| 失效链接 | 指向不存在的文件或章节 |
| 证据漂移 | “未确认”被改成“已确认” |

## 工作流

**增量编辑**：先搜重复，读目标文档和相关实现；范围不明确或删除可能损失业务知识时，先报告疑点再改，拿不准标 `[需确认]`。

**批量审计**：先列文档清单和修正方向；逐篇核对实现，只修漂移，不借机扩写或重构无关内容。是否分提交服从用户和项目约定，不自行 commit。

## 雷区

- 修漂移不等于重写文档。
- 不把项目细节写进通用规范，也不把通用常识塞进项目文档。
- 不静默重写 ADR 的实质决策；维护性修正和 supersede 规则见 `skill:decision-layering`。
